# `ash-nazg-dosbox-x` — engine container

A per-session DOSBox-X runtime container, spawned by the Ash Nazg
host shim when a user clicks **Run** on a DOS or Windows 3.x binary
in their Nextcloud Files. One container per active session;
destroyed on idle timeout (15 min default) or explicit close.

> **Status:** scaffold. The container builds and the binaries are
> present, but the entrypoint is a no-op. Real wiring lands in the
> `wire-dosbox-engine` change.

## Role in the architecture

```
┌──────────────────┐                   ┌─────────────────────────┐
│  Host container  │ ── spawn ──────► │  this engine container  │
│  (ash-nazg-host) │ ── kill ───────► │  (one per session)       │
└──────────────────┘                   │  WebDAV mount /mnt/files │
                                       │  KasmVNC stream :6901    │
                                       │  dosbox-x process        │
                                       └─────────────────────────┘
```

The host is the orchestrator and the only component that talks to
Nextcloud. This engine container is a **dumb runtime** — it knows
how to mount a filesystem, run DOSBox-X, and stream pixels. It does
not know what Ash Nazg is.

## Interface contract

These are the contract surfaces the wiring change will hook up. They
are **NOT** wired in this scaffold; they are documented here so the
contract is reviewable now.

### Environment variables (provided by host at spawn)

| Variable           | Required | Purpose                                                       |
|--------------------|----------|---------------------------------------------------------------|
| `NEXTCLOUD_URL`    | yes      | Base URL for the user's Nextcloud (used for the WebDAV mount).|
| `APP_TOKEN`        | yes      | Per-session AppAPI user token. Used as the WebDAV credential. |
| `USER_ID`          | yes      | Nextcloud user id; scopes the WebDAV mount to that user.      |
| `FILE_PATH`        | yes      | Files-relative path of the binary to execute (e.g. `/Programs/keen1.exe`). |
| `IDLE_TIMEOUT_SECONDS` | no   | Override the 900-second default; host normally fixes this.    |
| `KASMVNC_PASSWORD` | yes      | Per-session VNC password. Generated by the host, never reused. |

### Ports

| Port  | Protocol  | Purpose                                                  |
|-------|-----------|----------------------------------------------------------|
| 6901  | TCP / WS  | KasmVNC websocket. Host proxies this to the browser.     |

### Mount points

| Path         | Source            | Mode     | Notes                          |
|--------------|-------------------|----------|--------------------------------|
| `/mnt/files` | WebDAV (davfs2)   | rw       | The user's Files, scoped by `USER_ID` and `APP_TOKEN`. |
| `/tmp`       | tmpfs (256 MB)    | rw       | Provided by the host's run config; not declared here. |

### Resource posture

These limits are **set by the host** in `SessionConfig`, not by this
container. They are listed here so a reviewer can cross-reference
the engines spec and the actual `host/src/ash_nazg/engines/`
defaults that the wiring change will introduce.

| Resource         | Default  |
|------------------|----------|
| CPU              | 1.0 core |
| Memory           | 1024 MB  |
| Wall-clock idle  | 900 s    |
| Network          | none — host proxy network only |
| Root filesystem  | read-only |

## Building

```bash
# from repo root
docker buildx build \
    --platform linux/amd64,linux/arm64 \
    --build-arg KASMVNC_VERSION=1.4.0 \
    -t ghcr.io/mwest2020/ash-nazg-dosbox-x:scaffold \
    engines/dosbox-x/
```

`KASMVNC_VERSION` MUST match an upstream KasmVNC release tag at
<https://github.com/kasmtech/KasmVNC/releases>. The Dockerfile
fetches the matching `.deb` from the release; bumping the version
without verifying that the asset name still follows the
`kasmvncserver_bookworm_<version>_<arch>.deb` pattern will break the
build.

## Open architectural questions

These are deliberately parked here, not silently decided. They get
resolved in the `wire-dosbox-engine` change before the first real
end-to-end build is published.

### Q1 — KasmVNC install path: download `.deb` vs. use `kasmweb/core-debian-bookworm` base

**Current (this scaffold):** `FROM debian:12-slim`, then `curl` the
pinned `kasmvncserver_bookworm_<version>_<arch>.deb` from the
upstream GitHub release and `apt-get install` it.

**Alternative:** `FROM kasmweb/core-debian-bookworm:1.18.0` (Docker
Hub, maintained by Kasm). Pre-built, KasmVNC pre-installed, single
pinned image tag. Eliminates wget-fragility and the architecture-
conditional `.deb` filename logic, and is materially "boring
auditable"-er.

**Trade-off:** the pre-built image adds an external dependency on
Kasm's image-publishing pipeline (and a heavier base — the image
ships a window manager, X stack, and helper tooling we don't
necessarily need for DOSBox-X). The current `.deb` path keeps the
base minimal but inherits release-naming fragility.

**Decision pending in `wire-dosbox-engine`.** Don't preempt this in
intermediate changes; both paths are reachable from this scaffold
without rework.

### Q2 — KasmVNC `.deb` integrity verification

If Q1 lands on the `.deb` path, the wiring change MUST add at least
one of: pinned sha256, signature verification (KasmVNC publishes
detached signatures), or mirror-to-controlled-artifact-store. The
current `curl -fsSL` step trusts only HTTPS + GitHub's TLS, which
is below the project's "boring auditable" bar for shipped images.

## What is intentionally NOT here

- No DOSBox-X profile. The wiring change generates a per-session
  `.conf` based on the file being run.
- No bundled OS images, fonts, ROMs, BIOS files, or proprietary
  binaries. See `docs/bring-your-own-content.md` and the sandbox
  spec requirement "No bundled non-open-source content".
- No persistent state. Containers are ephemeral; everything that
  needs to survive a session lives in the user's Files via the
  WebDAV mount.
